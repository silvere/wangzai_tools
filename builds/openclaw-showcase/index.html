OpenClaw：
不是聊天玩具，
而是能真正替你干活的 AI 助手系统。

OpenClaw 把 AI 从“网页对话框”推进到“长期在线、能接消息渠道、能调工具、能存记忆、能做自动化、还能操作真实工作流”的层级。你可以把它理解为：一个可自托管的个人 AI 助手操作系统。

看常用命令 看 Skills 机制 看安全边界
多渠道接入飞书、Telegram、Discord、WhatsApp 等聊天入口
多工具文件、网页、命令行、设备、子代理、自动化都能挂进去
长期记忆支持 workspace、日记式记忆与长期记忆整理
真执行不是只会回答，而是会完成任务、产出文件、调用系统

为什么最近它火得快

1. 聊天入口自然直接在你平常用的 IM 里找 AI，不逼你换工作习惯。
2. 工作流能力强可以读文件、写文档、跑命令、接技能、做自动化。
3. 适合长期进化persona、memory、skills、heartbeat、子代理都能一起长。
4. 自托管可控你的上下文和配置在你自己机器上，不是全扔给闭源平台。

OpenClaw 到底是什么

一句话说透：把模型、工具、消息渠道、记忆和自动化编成同一套系统。

消息网关

OpenClaw 有自己的 Gateway，专门负责连接飞书、Telegram、Discord、WhatsApp、iMessage 等渠道。你不是在某个网页里“打开 AI”，而是在真实消息环境里长期跟它对话。

Agent 会话

它按聊天对象、群组、工作区路由不同会话。私聊和群聊、主会话和子会话可以隔离，避免上下文搅成一锅粥。

工具执行层

它不止回答，还能调用 shell、读写文件、抓网页、生成页面、转音频、操作设备、跑 ACP / 子代理等，属于“会动手”的 AI。

Messaging Gateway Workspace Memory Skills Cron / Heartbeat Subagents Tool Invocation Device Nodes

它怎么工作

不是“你问一句它答一句”，而是“接收任务 → 理解上下文 → 调工具 → 产出结果”。

① 接收输入来自飞书消息、群聊、附件、图片、语音、网页链接、命令触发器。
② 注入上下文读取 workspace 指令、persona、用户偏好、近期记忆、长期记忆、今天的事项。
③ 任务拆解判断是否要直接回答、是否要读文件、是否要开技能、是否要起子代理、是否要跑系统命令。
④ 调工具执行比如抓网页、转 Markdown、做页面、跑 GitHub、转语音、操作 Feishu 文档、调用本地程序。
⑤ 交付结果以文本、附件、页面、音频、GitHub Pages 链接、飞书文档等形式回传。

Skills：OpenClaw 最关键的扩展机制

Skill 不是插件市场噱头，而是把“某类任务怎么做”固化成可复用技能，让 AI 少走弯路。

Skills 能做什么

  • 定义某一类任务的标准流程
  • 告诉 Agent 什么时候该用某个能力
  • 约束怎么读写、怎么调用脚本、怎么处理失败
  • 把经验变成可复用的能力，而不是每次临场发挥

为什么它重要

  • 降低随机性：不是每次都瞎猜下一步
  • 提升成功率：同类任务路径更稳定
  • 方便扩展：新增一个 Skill，就像给系统长一个新器官
  • 便于沉淀经验：踩过的坑能写进技能，不用重复交学费

一个 Skill 通常包含什么

  • SKILL.md：说明用途、触发条件、执行方式
  • references/：补充参考资料
  • scripts/：实际可调用的脚本
  • EXTEND.md / 配置：用户偏好或默认参数
skills/ └── some-skill/ ├── SKILL.md ├── scripts/ └── references/

常见高价值 Skill 类型

  • 网页抓取 / URL 转 Markdown
  • 文章翻译 / Markdown 转 HTML
  • GitHub / Issues / PR 操作
  • 本地 Whisper 转写
  • TTS 语音生成
  • 图像生成 / 封面图 / 幻灯片 / 信息图
  • YouTube / WeChat / Feishu / Drive 系列能力

什么时候该开 Skill

  • 任务有明显专业流程时
  • 有固定工具链和文件规范时
  • 失败代价较高，需要减少随机性时
  • 同一类事你会反复做时
装太多没经过筛选的 Skill，只会让系统越来越乱。真正有价值的是：高频、稳定、对你工作流有帮助的那几类。Skill 选型本质上是工具治理，不是收集癖。

安全：OpenClaw 真正的门槛，不是安装，而是边界

能力越大，越不能拿“能跑起来”当胜利。你真正要管的是权限、凭据、外发、网络暴露、自动化范围。

核心安全原则

✓
最小权限
只开需要的工具、渠道和插件，不要一上来全开。
✓
凭据隔离
Token / Cookie / Secret 放私有目录，别塞进公开 workspace 文档里。
!
外发前确认
邮件、社媒、共享链接、公网暴露都要把风险想清楚。
✕
别把生产环境当试验田
不熟的技能、脚本、自动化别直接在关键机器上乱试。

最常见的风险点

  • 把高敏感 token / cookies 发进聊天或写进公开文件
  • 为了图快，把本地服务直接暴露到公网
  • 默认允许所有联系人/群聊调用高权限工具
  • 没审过第三方 Skill 就直接装
  • 让 Agent 在没有审计和限制的情况下长期跑自动化
  • 先只接一个私聊渠道
  • 把 exec / 浏览器 / 外发能力分阶段开启
  • 优先用本地文件、附件、GitHub Pages 等受控方式交付
  • 高敏感凭据单独存放，权限 600
  • 定期跑状态检查和审计
因为 demo 看起来都很丝滑：一句话安装、几分钟接通、马上能用。但真正让你后悔的，通常不是“功能太少”，而是“权限开太大、凭据管太松、自动化跑太野”。OpenClaw 最好玩的地方，往往也是最该谨慎的地方。

常用命令：给新手一份真正有用的命令表

不是把 help 全贴一遍，而是把最常用、最容易出事、最值得记的命令讲清楚。

先看系统是不是活着、渠道是不是通、会话有没有问题：

openclaw status openclaw status --all openclaw status --deep

用途：检查本地状态、渠道、会话、队列、故障线索。
什么时候用：出故障第一反应就该跑这个。

openclaw gateway status openclaw gateway start openclaw gateway stop openclaw gateway restart

用途：控制消息网关服务。
注意：改完关键配置后，通常要 restart 才会真正生效。

openclaw config get channels.feishu.streaming openclaw config set channels.feishu.streaming true openclaw config set tools.profile "coding"

用途：查询或修改 OpenClaw 配置。
注意：很多配置是“写入成功 ≠ 立即生效”，要记得配合 gateway restart。

openclaw channels login

用途：登录或重新建立消息渠道连接（如 WhatsApp Web）。

openclaw logs --follow

用途：盯实时日志。
适合：刚改完配置、刚重启、某个渠道就是不回消息的时候。

openclaw gateway probe

用途：检查 reachability 和连接层面的问题。适合“服务看着活着，但就是连不上”的情况。
