智能体的自适应推理时扩展

Agentic Test-Time Scaling for WebAgents (CATTS)
Nicholas Lee, Lutfi Eren Erdogan, Chris Joseph John, Surya Krishnapillai, Michael W. Mahoney, Kurt Keutzer, Amir Gholami
2026-02-12 · cs.AI, cs.CL · arXiv · PDF
💡 一句话总结：通过投票分布的不确定性（熵和 margin）来动态分配推理计算资源，只在决策真正有争议时才增加采样，实现更高效的 Web Agent 推理时扩展。

🎯 研究动机与背景

Test-time scaling（推理时扩展）已成为提升 LLM 性能的标准方法——在推理时花更多计算来生成更多 token，而非在预训练时投入更多。对于单步推理任务，多次采样 + 多数投票能带来显著提升。

但在多步智能体任务中，情况完全不同：

  • 错误累积效应：每一步的小错误会在长链路中不断放大，一个错误决策可能导致整个轨迹不可恢复
  • 均匀扩展快速饱和：在每一步都增加相同的采样数量，收益递减明显
  • 简单步骤浪费计算：大多数步骤是显而易见的（如点击提交按钮），对这些步骤做 test-time scaling 纯属浪费
  • 仲裁器的双刃剑：引入额外 LLM 做决策选择虽能提升平均表现，但可能推翻正确的高共识决策

核心问题：如何在多步智能体任务中，智能地分配推理时计算资源？

🔬 核心方法：CATTS

整体框架

CATTS（Confidence-Aware Test-Time Scaling）的核心思想非常简洁：用投票分布的不确定性来决定是否需要额外计算。

Step 1: 动作采样与聚类

在每个时间步 t，从基础模型采样 N 个候选动作，然后用语义去重将它们聚类为等价组，得到投票分布 p_t(a)。

Step 2: 不确定性度量

计算两个关键统计量：

  • 熵 H_t：衡量整体分歧程度，熵越高说明投票越分散
  • Margin Δ_t：第一名和第二名得票率之差，差距越大说明共识越强

Step 3: 条件性仲裁

只有当不确定性超过阈值时（高熵或低 margin），才调用仲裁器（Arbiter）进行更深层的推理选择；否则直接用多数投票结果。

关键洞察

论文发现成功轨迹和失败轨迹的不确定性特征有显著差异：成功任务全程保持低熵（~0.3）和高 margin（~0.7），而失败任务的熵更高、margin 更低，且在后期步骤差距更大。

高共识覆盖的危害

当仲裁器在高共识步骤（Δ_t > 0.7）推翻多数投票时：

  • 无覆盖的任务成功率：46.9%
  • 有覆盖的任务成功率：35.0%
  • 差距：11.9%（p=0.026，Fisher 精确检验）

💡 创新点与贡献

  • 首次系统研究多步智能体任务中的推理时扩展行为
  • 发现投票分布的不确定性（熵和 margin）与任务成功率强相关
  • 提出简洁的条件性计算分配策略，无需访问 token 级概率
  • 揭示了仲裁器覆盖高共识决策的危害，量化了"过度思考"问题
  • 方法模型无关，适用于任何支持采样的 LLM API

⚠️ 局限性与未来方向

  • 仅在 Web 导航任务上验证，尚未扩展到其他智能体领域（代码生成、OS 操作等）
  • 阈值 τ 需要手动调节，未来可探索自适应阈值
  • 语义去重依赖额外 LLM 调用，增加了一定开销
  • 未探索与 RL 训练方法的结合
  • 基础模型固定为 gpt-oss-120b，不同模型可能表现不同

🛠️ 实践启示

  • 不要盲目增加采样：均匀扩展在多步任务中收益递减，要智能分配
  • 监控投票分布：熵和 margin 是判断决策质量的有效信号
  • 谨慎使用仲裁器：在高共识情况下，额外推理反而有害
  • 关注计算效率：CATTS 用更少 token 获得更好效果，对降低部署成本有直接意义
  • 适用于生产环境：方法简单、可解释、不需要模型内部访问权限

Q1: CATTS 和 DeepConf 有什么区别？ ▼

核心区别在于不确定性信号的来源。DeepConf 使用 token 级别的对数概率来计算置信度，需要访问模型内部的 logprobs；而 CATTS 仅使用投票分布（多次采样后的动作聚类分布）来计算熵和 margin，不需要任何模型内部访问。这意味着 CATTS 可以用于任何支持采样的 API（包括闭源模型），而 DeepConf 受限于提供 logprobs 的模型。在效果上，两者表现相当，但 CATTS 的适用范围更广。

Q2: 为什么均匀扩展在多步任务中效果不好？ ▼

主要有两个原因：1）大多数步骤是「简单步骤」（如点击按钮、填写表单），模型已经很确定正确答案，额外采样只是浪费计算；2）在「困难步骤」中，投票可能分散在多个看似合理的选项上，简单的多数投票无法有效区分。这导致了一个悖论：简单步骤不需要额外计算，困难步骤又无法通过简单增加采样来解决。CATTS 的解决方案是只在困难步骤投入额外计算，并使用更强的选择机制（仲裁器）。

Q3: 仲裁器（Arbiter）具体是怎么工作的？ ▼

仲裁器本质上是一个额外的 LLM 调用。在采样 N 个候选动作并聚类后，构建一个包含当前页面状态和所有候选动作（每个聚类一个代表）的 prompt，让 LLM 从中选择最佳动作。论文中仲裁器使用与 agent 相同的模型（gpt-oss-120b）。还可以进一步做「仲裁器扩展」：调用 K 个独立的仲裁器，对它们的选择再做多数投票。

Q4: 熵和 margin 这两个指标哪个更有效？ ▼

论文发现两者都有效且互补。熵衡量的是整体分歧程度（投票分布的信息量），margin 衡量的是决策的确定性（第一名和第二名的差距）。在实践中，高熵+低 margin 表示「困难决策」，低熵+高 margin 表示「简单决策」。CATTS 可以使用任一指标或两者结合来设定阈值。论文中的分析显示，成功轨迹全程保持低熵（~0.3）和高 margin（~0.7），这个特征非常稳定。

Q5: 这个方法能用在非 Web 的智能体任务上吗？ ▼

理论上完全可以。CATTS 的核心思想——用投票分布的不确定性来动态分配计算——是通用的，不依赖于 Web 导航的特定特征。任何多步决策任务（代码生成、OS 操作、游戏 AI 等）都可以应用类似的策略。但论文目前只在 WebArena-Lite 和 GoBrowse 上验证，其他领域的效果需要进一步实验。不同领域的动作空间和错误传播模式可能不同，阈值也需要重新调整。

Q6: 语义去重是怎么做的？为什么需要它？ ▼

在 Web 智能体中，不同的文本输出可能表示相同的语义动作（比如 'N/A' 和 'Not found'，或者 'click button Submit' 和 'click the submit button'）。如果直接做字符串匹配的多数投票，这些语义相同但文本不同的动作会被当作不同选项，导致投票分散。论文使用一个轻量级 LLM 来将候选动作聚类为语义等价的组，然后在聚类级别做投票。这是准确计算投票分布的前提。

Q7: CATTS 的计算开销如何？值得在生产环境使用吗？ ▼

CATTS 的额外开销主要来自：1）多次采样候选动作（N 次 LLM 调用）；2）语义去重（1 次轻量 LLM 调用）；3）条件性仲裁（仅在不确定步骤调用）。相比均匀扩展（每步都做 N 次采样 + 仲裁），CATTS 节省了 2.3 倍 token。对于生产环境，这意味着更低的 API 成本和更低的延迟，同时获得更好的成功率。方法简单、可解释、易于实现，非常适合生产部署。

Q8: 论文中的「过度思考」问题是什么？ ▼

「过度思考」（Overthinking）指的是在模型已经很确定正确答案的情况下，额外的推理步骤反而导致错误决策。具体表现为：当候选动作的投票分布高度集中（比如 9/10 的候选都选择同一动作），仲裁器可能会推翻这个高共识决策，选择一个少数派动作。论文量化了这个问题：无高共识覆盖的任务成功率为 46.9%，有覆盖的仅为 35.0%，差距 11.9%。这说明「更多计算 ≠ 更好结果」，计算分配的位置比总量更重要。

Q9: 这篇论文和 OpenAI 的 o1/o3 系列有什么关系？ ▼

都属于 test-time scaling 的大方向，但侧重点不同。o1/o3 系列主要通过内部的 chain-of-thought 推理来扩展单步推理能力，是「深度」扩展；而 CATTS 关注的是多步智能体任务中的「宽度」扩展——在每一步采样多个候选并智能选择。两者可以互补：可以用 o1/o3 作为基础模型，再在上层应用 CATTS 的动态计算分配策略。论文的贡献在于揭示了多步任务中推理时扩展的独特挑战和解决方案。

Q10: WebArena-Lite 和 GoBrowse 是什么样的基准？ ▼

WebArena-Lite 是 WebArena 的精简版，包含 165 个 Web 导航任务，涵盖电商、论坛、内容管理等场景，使用程序化的成功检查（不依赖 LLM 判断）。GoBrowse 是一个更大的基准，包含 341 个任务，使用 LLM-as-a-judge（Qwen3-VL-30B）来评估任务完成度。两个基准都要求 agent 在真实的 Web 环境中执行多步操作，如搜索、填表、导航等。

Q11: 如果我想复现这篇论文的方法，需要什么？ ▼

你需要：1）一个支持多次采样的 LLM API（不需要 logprobs 访问）；2）一个 Web 智能体框架（如 ReAct）；3）语义去重的 LLM（可以用较小的模型）；4）WebArena 或类似的评测环境。核心实现并不复杂：在每步采样 N 个候选 → 语义聚类 → 计算熵和 margin → 如果超过阈值则调用仲裁器，否则用多数投票。阈值可以在小规模验证集上调优。

Q12: 这个方法对不同的基础模型效果一致吗？ ▼

论文主要使用 gpt-oss-120b 作为基础模型，没有在多个模型上做对比实验。理论上，CATTS 的核心假设——投票分布的不确定性与任务成功相关——应该对大多数模型成立，因为这是一个统计性质而非模型特定的。但不同模型的最优阈值、最优采样数 N 可能不同。更强的模型可能在更多步骤上达成高共识，需要更少的仲裁介入；更弱的模型可能需要更多的计算分配。
